BoardRipper

Web-based PCB boardview viewer for board-level repair. Free, AGPL-3.0.

Latest release: v0.7.0 — released 2026-05-05

BoardRipper opens eleven boardview formats and the matching PDF schematic, side by side, in a browser. Designed to be self-hosted on a NAS via Docker. Works on desktop and on touch screens.

It is the viewer only. BoardRipper does not ship with any boardview files, schematics, or PDFs — you bring your own. The bundled board database is reference metadata (board numbers and device models), no copyrighted content.

Click for full size. Left to right: a single board view, a multi-layer board with the layer sidebar, a board with multiple revisions grouped under the Revisions tab, and the Library panel sorting boards by parsed metadata and filenames.

What is this, and why self-host it?

Docker is a way to ship an application as a single image that runs the same on any machine: no installer, no per-OS binary, no dependency hell. You give Docker the image, point a port at it, and it runs. BoardRipper ships as a ~5 MB Docker image; docker compose up -d and it's online.

Why run it on a NAS or shop server:

  • One library, every workstation. Boards and schematics live in one place — open them from the bench laptop, the office desktop, or a tablet next to the microscope.
  • No per-machine install. It's a web page. Send a coworker the URL and they're up.
  • You own the data. Boards stay on your hardware, nothing leaves the network.
  • Touch-friendly. Pinch-zoom on a shop tablet works the same as the mouse on a workstation.
  • One-click container update from the toolbar.

Runs on

Server side — Docker on Linux, macOS, or Windows; or the standalone Go binary (Linux x86_64, macOS Intel/ARM, Windows x64). Runs happily on a Synology or QNAP NAS, a Raspberry Pi 5, an old laptop in a closet, or anything else that can run a container.

Client side — any modern browser (Chrome, Firefox, Safari, Edge) on any OS, including iPad and Android tablets. There’s no per-client install — you open the URL.

Old hardware — the rendering pipeline is GPU-accelerated WebGL, which leans on the integrated GPU rather than the CPU. Bench-tested on 3rd- and 4th-generation Intel laptops (HD 4000 / HD 4400 / HD 4600 era), frame rate sits in the 10–30 fps range depending on the board — usable but visibly slower than a modern GPU. On those machines you may want to lower the browser zoom or window size and turn off a few of the visual extras (label tier, glyph overlay) in the Settings panel; the Debug panel surfaces frame-time so you can see what’s slow.

Features

  • GPU-accelerated rendering — 10,000+ components at 60 fps on a modern GPU (PixiJS / WebGL).
  • Multi-board tabs — open several boards at once, switch between them.
  • Multi-layer support — show / hide top, bottom, and inner layers independently. Butterfly mode shows top and bottom side by side.
  • Net highlight — click a pin and the whole net lights up across the board. A chain-adjacent mode walks through neighbouring nets too.
  • PDF schematic, in sync — pan, zoom, search, night mode. Right-click a net to search it in the schematic.
  • Touch input — pinch-zoom, drag-pan, tap to select. Usable on a tablet at the bench.
  • Customizable colors — per-net rules by name pattern (e.g. GND, VCC, PP*), per-component-type fills (R / C / L / U / Q / D / J), per-layer colors, label / pin / outline tuning. Live preview before applying.
  • Multi-line search — each line is an extra AND-filter, useful when a component carries value and voltage on separate label lines: 
    20uF
10V
C12*
    finds capacitors that match all of: 20µF, 10 V, designator starting with C12.
  • Drag-and-drop import — drop a board file or a folder onto the window and it loads. To make dropped files persistent across container restarts, mount one writable folder at /library/incoming (see Docker setup). Without it, drops live only in the browser’s IndexedDB cache.
  • Board library — scan folders, browse by board number or model. Filenames are auto-sorted: board numbers, manufacturers, and revisions are parsed automatically.
  • Board database with heuristics — a bundled reference DB maps board numbers to manufacturer / ODM / laptop or phone model and groups boards by device.
  • PDF auto-link — boards and their schematics get matched up automatically when both are present in the same library.
  • IndexedDB cache — re-open without re-parsing.
  • Self-update — one-click signed update from the toolbar. No GitHub token required; updates are signed with an offline key and verified by the running container before install.

Supported formats

BVR1.bvr / .bvTab-delimited boardview export, coords ×1000 → mils
BVR3.bvr / .bvKeyword-value boardview export, relative pin coords
BRD.brdApple / Mac repair, bit-rotation obfuscated binary
BDV.brd / .bdvPlain-text boardview (BRDOUT / NETS / PARTS / PINS / NAILS)
BDV ASC.bdvHonhan / Tebo-ICT, line-key cipher
FZ.fzASUS, RC6-encrypted, zlib-compressed
GenCAD.cadGenCAD 1.4 PCB interchange (text)
Mentor Neutral.cadMentor Boardstation neutral file (Samsung / Quanta / Compal exports)
XZZ.pcbXZZ PCB, DES-encrypted boardview
TVW.tvwTeboview binary, multi-layer + traces + drill data
Cadence Allegro BRD.brdCadence Allegro PCB binary, v16.x / v17.x / v18.x. v15 partial, still in beta.

Both .cad and .brd are shared extensions — BoardRipper sniffs file content to pick the right parser.

Docker setup

Typical deployment on a NAS or small home server:

services:
  boardripper:
    image: ghcr.io/alexeyinwerp/boardripper:latest
    ports:
      - "8081:8080"
    volumes:
      - ./data:/data
      # Your library: mount EVERYTHING read-only (:ro) by default.
      - /path/to/MacBooks:/library/MacBooks:ro
      - /path/to/iPhones:/library/iPhones:ro
      # The ONLY writable mount the container should ever see:
      - /path/to/incoming:/library/incoming:rw
      - /var/run/docker.sock:/var/run/docker.sock
    environment:
      - PORT=8080
    restart: unless-stopped

Then: docker compose up -d → open http://your-nas:8081

Mount your board folders as subdirectories of /library — each one shows up as a top-level folder in the Library panel.

Keep every other mount :ro. /library/incoming is the only path BoardRipper should be able to write to — that’s where dropped files land. The container does not need write access to your existing board folders, your schematics, or anything else.

Self-update needs the /var/run/docker.sock mount — that's what lets the container pull the new image and restart itself when you click "update" in the toolbar. Remove it if you'd rather update manually with docker compose pull && docker compose up -d.

Download

Two sources, same signed bytes. Pick whichever your network prefers:

  • Container registry (recommended) — docker pull ghcr.io/alexeyinwerp/boardripper:latest. Public, no login. Fast incremental updates via layer dedup.
  • Tarball mirror — latest.tar.gz (OCI image archive, ~5 MB), shipped alongside a signed manifest.json. Use this if your network blocks container registries; load with docker load < latest.tar.gz.
  • Older versions — browse the release archive.
  • Release notes and version history — changelog.

Both downloads ship the same image. The running container verifies an Ed25519 signature against a key compiled into its own binary before installing any update — a hijacked mirror cannot deliver a forged update.

Drop-to-update fallback

For installs that can’t reach the registry or the mirror (firewalled networks, air-gapped shops, or recovering from a broken self-update), download the bundle and drop it onto the BoardRipper window:

  • latest-update.tar — signed bundle (manifest + signature + image), ~30 MB

Drag the file anywhere on the BoardRipper UI. A confirmation prompt appears; click OK and the running container verifies the signature, applies the update, and restarts. The browser reloads after ~30 s. Same trust envelope as the in-app updater — the manifest signature is what grants trust, the file itself is treated as untrusted bytes until verification passes.

About this software

I have a programming background, but BoardRipper is not hand-written. The whole codebase — parsers, renderer, panels, backend — was generated with Claude Code, working from format specs and reference implementations from the boardview community. I review, test, and direct it; the lines of code came out of an LLM.

For context: I’m a father of two and I work 50–60 hours a week. Hand-writing a project of this scope was never going to happen — this was the only way it was ever going to exist. Sceptics will dismiss anything in this category as “AI slop”; I think BoardRipper is a fair counterexample. Eleven boardview parsers, ten thousand components rendered at 60 fps, and not one line of code written by hand. It works on real boards in real shops. That’s the answer I have for the slop critique — not arguing about it, just shipping something useful.

Treat that as you would any AI-assisted software: it works on the boards I’ve tested, it has bugs I haven’t found yet, and the right level of trust is “a useful viewer, not a forensic tool.” Bug reports welcome.
