# Release archive — maintenance runbook

The version archive lives at <https://www.ripperdoc.de/boardripper/archive.html>.
It's the canonical "give me an older version" landing page. The GitHub
Releases tab intentionally shows only the current release — every prior
release was deleted during the 2026-05-13 cleanup, and new releases will
be pruned from that surface as they retire too.

This doc covers:

1. What the archive contains and how it's regenerated
2. The retention policy: where each version lives (GHCR / FTP / source-only)
3. Routine maintenance — what to do when cutting a new release
4. Retiring a version — pruning GHCR while keeping the tarball discoverable
5. Permanently burying a version — when compliance / security demands it

## What the archive contains

Every entry in `CHANGELOG.md` (one row per `## vX.Y.Z` heading) shows up
as a row, sorted newest → oldest. Each row carries up to four download
paths and a source link:

| Column | Source of truth | Notes |
|---|---|---|
| Tarball | `https://www.ripperdoc.de/boardripper/releases/boardripper-vX.Y.Z.tar.gz` | The OCI image archive, `docker load`-able. Mirrored for every v0.19.0+ release. |
| Drop-bundle | `https://www.ripperdoc.de/boardripper/releases/boardripper-update-vX.Y.Z.tar` | Signed manifest + signature + tarball, packaged as a single `.tar` for the drag-onto-window recovery path. Only emitted from v0.19.4+ (the bundle format didn't exist before). |
| GHCR | `ghcr.io/alexeyinwerp/boardripper:vX.Y.Z` | A handful of versions are retained for direct `docker pull`. Everything else is pruned (see Retention below). |
| Source | `https://github.com/AlexeyInwerp/BoardRipper/tree/vX.Y.Z` | Always available. Git tags are immutable — never delete them. |

The page is generated by `scripts/release/gen-archive.sh`. It reads
`CHANGELOG.md` and emits HTML based on hard-coded availability rules:

- `HAS_TARBALL` regex (currently `^v0\.(19|20)\.`): which versions are
  expected to have a `.tar.gz` on FTP. Pre-v0.19 used a now-retired GH
  Actions pipeline and was never mirrored.
- `HAS_BUNDLE` regex (currently `^v0\.(19\.[4-9]|20\.)`): which versions
  have the signed `.tar` drop-bundle. v0.19.0–v0.19.3 predate the bundle.
- `GHCR_KEEP` env var (default `v0.19.0 v0.19.5 v0.20.5 v0.20.8`): which
  versions are still present on GHCR after the retention sweep.

If you change the retention policy, edit `GHCR_KEEP` either in the script
itself or by exporting it before invoking `release.sh`.

## Retention policy

Three tiers. Versions migrate downward over time; never upward.

### Tier 1 — Current
**One version.** The one whose digest is in the signed `manifest.json` on
FTP. Always pull-able from GHCR by both tag (`:vX.Y.Z`) and `:latest`.
GitHub Releases has exactly one entry pointing at it.

### Tier 2 — Kept on GHCR
**Three or four hand-picked older versions** (the "few middle" — currently
v0.19.0, v0.19.5, v0.20.5). Direct `docker pull` works. Useful for:

- Downgrade testing during regression hunts
- Customers who want to pin a specific version
- Forensic debugging of a known-bad upgrade path

The selection should span the active release range. Rough rule of thumb:
keep the oldest still-on-GHCR (the "epoch" — currently `v0.19.0`, the
bridge release that introduced the new pipeline) plus one or two
intermediate milestones.

### Tier 3 — Archive-only
**Everything else.** GHCR tag deleted via the GitHub Packages API. The
tarball and drop-bundle stay on FTP at `releases/boardripper-vX.Y.Z.tar.gz`
+ `releases/boardripper-update-vX.Y.Z.tar`. Discoverable through the
archive page; not discoverable through `docker pull`.

This is the steady state for most historical versions. Storage cost on
FTP is small (~7 MB per image tarball, ~13 MB per drop-bundle).

### Tier 4 — Source-only
**Pre-v0.19.0 releases.** No tarball, no bundle, no GHCR. Only the git
tag exists. The archive page lists them with a source-only link to
`github.com/AlexeyInwerp/BoardRipper/tree/vX.Y.Z`.

This is also where Tier-3 versions go if FTP storage becomes a problem
or if a version needs to be **permanently buried** for compliance reasons
(see below).

## Routine maintenance (every release)

`scripts/release.sh` regenerates and uploads `archive.html` as part of
the FTP push. No manual step needed for the common case — the newly-cut
version appears at the top of the table on the next release.

That's it. The archive update is folded into the normal release flow.

## Retiring a version (Tier 2 → Tier 3)

When a previously-pinned version no longer needs to be on GHCR:

1. Edit `GHCR_KEEP` in `scripts/release/gen-archive.sh` to drop the
   version. (Or override via env on the next release.)
2. Delete the GHCR tag:
   ```bash
   # Get a PAT with read:packages + delete:packages scope from
   # https://github.com/settings/tokens/new
   GH_TOKEN=ghp_xxx gh api -X GET \
     /users/AlexeyInwerp/packages/container/boardripper/versions \
     --paginate --jq '.[] | select(.metadata.container.tags[] == "vX.Y.Z") | .id'
   # → some-id
   GH_TOKEN=ghp_xxx gh api -X DELETE \
     /users/AlexeyInwerp/packages/container/boardripper/versions/<some-id>
   ```
3. Cut a release (or run `scripts/release/gen-archive.sh` standalone +
   `lftp`-upload `archive.html` separately) so the table reflects the
   change.

The version's `.tar.gz` and update bundle on FTP are unchanged. Anyone
who needs the image can still `docker load < boardripper-vX.Y.Z.tar.gz`.

### Why we don't auto-prune

Retention is a judgment call that depends on user needs (do we still
need to test downgrade *from* this version? Is anyone known to be
running it in production?). Better to keep it explicit and audit-trail
each retirement than to expire on a fixed clock.

## Permanently burying a version (Tier 3 → Tier 4)

Only do this when compliance or security demands it — e.g. a version
contained leaked credentials, the wrong license header, or a vendor
asked for takedown.

1. Delete the GHCR tag (see above).
2. Delete the tarball and drop-bundle from FTP:
   ```bash
   lftp -u "$FTP_USER,$FTP_PASSWORD" ftp.ripperdoc.de <<LFTP_EOF
   set ftp:ssl-allow no
   cd /public_html/boardripper/releases
   rm -f boardripper-vX.Y.Z.tar.gz
   rm -f boardripper-update-vX.Y.Z.tar
   bye
   LFTP_EOF
   ```
3. Update `HAS_TARBALL` / `HAS_BUNDLE` in `gen-archive.sh` to exclude
   the version (or add a special-case skip — for one-off cases an
   inline exception is fine).
4. Regenerate and re-upload `archive.html` so the row falls back to
   source-only.
5. **Do not delete the git tag.** Tags are immutable history; deleting
   one rewrites the public record and breaks anyone with a cloned repo.
   If the *code* is the problem, file a CHANGELOG-level note pointing
   at the replacement commit instead.

## Tooling reference

- `scripts/release/gen-archive.sh` — emits `archive.html` from
  `CHANGELOG.md`. Standalone-runnable:
  ```bash
  REPO_ROOT=$(pwd) OUT=/tmp/archive.html ./scripts/release/gen-archive.sh
  ```
- `scripts/release/site-artifacts.sh` — calls `gen-archive.sh` as part
  of the full site-render. Invoked by `scripts/release.sh`.
- `scripts/release.sh` — uploads `archive.html` to FTP alongside the
  manifest + tarball.

## Open improvements

- The `HAS_TARBALL` / `HAS_BUNDLE` regexes are hard-coded; ideally they'd
  HEAD-probe FTP for each version and reflect reality. Today they
  encode the expected state from when each format was introduced — good
  enough until a bury operation falsifies the assumption.
- The archive page doesn't surface known bugs in each release. A future
  enhancement: parse the `## Important` section of each CHANGELOG entry
  and show a warning glyph in the row when present.
