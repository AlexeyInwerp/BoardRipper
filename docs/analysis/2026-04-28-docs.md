## New Findings (Current Code State)

### Critical — Process & Dependency Tracking

#### DOC-NEW-1: Feature Spec Count Explosion (No Tracking)
**Severity:** IMPORTANT  
**Files:** `docs/superpowers/specs/`, `docs/superpowers/plans/`  
**Metric:** 70 spec files vs 17 plan files

The `docs/superpowers/` directory has accumulated:
- **70 `.md` files** under `specs/` (2026-03-18 through 2026-04-27)
- **17 `.md` files** under `plans/` (same date range)
- **Ratio:** 4.1 specs per plan

**Evidence:**
```bash
find docs/superpowers/specs -name "*.md" | wc -l  → 70
find docs/superpowers/plans -name "*.md" | wc -l  → 17
```

**Issue:** Each spec is a detailed design document. With only ~24% of specs having a corresponding plan, there's a silent risk that design work happens without an implementation checklist. The prior code review expected plans-to-specs as a 1:1 pairing (with rare exceptions). Current ratio suggests either:
1. Many designs never implemented (museum docs)
2. Plans written retrospectively or embedded in commits
3. Multiple specs per plan (valid but undocumented)

**Risk:** Months from now, a developer picks up a spec and discovers no plan, or a plan with a different date range. Documentation becomes a liability instead of a guide.

**Correction:** Add a `docs/superpowers/STATUS.md` matrix:
```
| Spec Date | Title | Plan Date | Status | Commits |
| 2026-04-27 | binding-categorization-design | 2026-04-27 | In Progress | — |
| 2026-04-27 | library-history-favorites-design | none | Design-only? | — |
| ...
```

Every spec and plan is a row. Track: "Has Plan", "Implemented", "Merged", "Deployed". Update on each release cycle.

**Impact:** Makes spec→plan→commit traceability explicit. Prevents museum-doc accumulation.

### Important — Code Comment Quality

#### DOC-NEW-4: Inline Comments Explain WHAT, Not WHY, in Two Key Modules
**Severity:** MINOR (Quality debt)  
**Files:** 
- `src/backend/databank/db.go` (100 lines, 8 comments) — 8% comment density
- `src/frontend/src/renderer/BoardRenderer.ts` (3100+ lines, 40 comments) — 1.3% density

**Analysis:**
Both files have comments, but pattern analysis shows a mix of good and noise:

**Good comments (WHY):**
```go
// DB wraps a SQLite connection with databank-specific helpers.
// Uses separate read and write connection pools so readers never block on writes
// (WAL mode allows concurrent reads alongside a single writer).
```

```typescript
// Pre-scan to detect whether PIN_ORIGIN values are relative to PART_ORIGIN
// or already absolute. Many real-world BVR3 files (e.g. Apple boards) store
// absolute pin positions even when PART_ORIGIN is non-zero. We detect this
// by looking at parts whose PART_ORIGIN is non-zero: if every such part's
// first PIN_ORIGIN exactly equals its PART_ORIGIN the file uses absolute
// coordinates and we must NOT add the origin offset.
```

**Noise (WHAT — restates code):**
```typescript
// Hide-text-during-zoom: detect actual zooming via per-frame scale comparison
// Selection blink state (triggered by focusPart / PDF reverse search)
// Pin labels raised above the ambient dim overlay for the selected part.
```

These describe the variable purpose but don't explain non-obvious design choices.

**Pattern:** Older, stable code (parsers, Go backend) has excellent "WHY" comments. Newer rendering code (BoardRenderer) has adequate "WHAT" comments but sparse "WHY" for complex logic.

**Correction:** Not actionable as a single task; document as a code review guideline in CLAUDE.md:
```markdown
## Comments
- Focus on WHY, not WHAT. The code shows WHAT; comments explain architectural intent.
- For non-obvious algorithms (e.g., coordinate system conversions, heuristics), add 2–3 lines.
- Variable declarations: comment only if the name + type don't clearly state purpose.
```

**Impact:** Maintainability guideline, helps future reviewers.

## Cross-Cutting Observations on the Doc System

### What's Working Well

1. **Format specifications** are exemplary — thorough, well-structured, consistent prose style. TVW_FORMAT.md (500+ lines) is an outstanding reference for a complex binary format. BVR_FORMAT.md, BDV_FORMAT.md are authoritative.

2. **CLAUDE.md** is accurate and actionable. The PixiJS destroy warning (lines 84–91) is valuable institutional knowledge properly preserved. Recent additions (PDF pipeline, scroll modifier zoom speeds) are well-documented.

3. **Feature specs (2026-04) are high-quality.** Both binding-categorization and library-history specs follow the same structure: Goal → Non-Goals → User Stories → Design → Affected Files → Edge Cases → Testing. Implementation plans are detailed checklists with step-by-step tasks.

4. **Inline code comments are improving.** BVR side inversion is now documented in two parser files. BDV shoelace behavior documented. BVR3 absolute-vs-relative pin detection has a 9-line block comment explaining the heuristic.

5. **README.md is accurate and user-centric** — covers Docker, standalone, desktop, Synology NAS setup with proper port numbers and env var usage.

### What Needs Attention

1. **Spec proliferation without status tracking.** 70 specs in `docs/superpowers/specs/` but no matrix tracking which have plans, which are merged, which are deployed. Risk: a maintainer in 6 months picks up spec X and wonders if it was built.

2. **Cross-spec dependencies are implicit.** Binding-categorization spec depends on library-rework, but the dependency is a footnote in an older spec, not a prerequisite in the plan.

3. **Feature plan / spec dates don't always align.** E.g., binding-categorization spec + plan both dated 2026-04-27, but library-rework spec is 2026-04-22. The relationship is documented but requires reading both specs to understand.

4. **Agent infrastructure docs (docs/agents/)** are present but not audited by this review. The 9 specialist agents listed in PLANNING.md have FILE_MAP.md and MEMORY.md but no automated sync between code changes and doc updates (per PLANNING.md note: "Agent infrastructure: mechanize FILE_MAP refresh + WORK_LOG entries so docs don't drift silently across many commits.").

5. **Museum docs risk.** No policy for archiving outdated specs. Example: 70+ files in superpowers/specs, but is `2026-03-18-pdf-glyph-debug-optimization-design.md` deployed, abandoned, or in-progress? No status visible.
