# Test Coverage & Quality Review — BoardRipper v0.14.0 (2026-04-28)

## Executive Summary

BoardRipper has **23 Playwright E2E specs** (5018 lines) covering UI flows, parser integration, and multi-format stability. **Backend Go tests** exist but are minimal (3 test files, ~17 unit tests). Coverage is **unevenly distributed**:

- **Strong:** v8 binding-categorization API (16 tests), net-lines 3-state toggle, layer switching, panel UI
- **Weak:** Parser unit tests (only BVR1, BDV, TVW have E2E specs; BRD/BVR3/FZ/CAD/XZZ entirely absent), PDF viewer flows (basic smoke but no zoom/tier/watermark edge cases), backend handlers (only file ops, no databank/board API endpoints)
- **Missing:** Pure utility function tests (no vitest/jest), visual regression, performance baselines, history-favorites flow, real-pad-shape rendering coverage

**Test quality issues:** Hard-coded board IDs (873, BOARD_ID) fragile against DB changes, excessive `waitForTimeout` (100+ instances) over proper `waitFor`, CSS selector brittleness, no isolation/cleanup in some E2E tests, WebGL adapter warnings unhandled in some suites.

**Recommendations:** Unit test framework for parsers, fixture-based board/PDF selection, remove config hardcoding (ports 1336/8083/5174), comprehensive PDF viewer test suite, backend API endpoint coverage.

## Critical Findings

### 1. Hard-Coded Resource IDs Fragile Against DB Changes
**File:** `src/frontend/tests/binding-categorization.spec.ts:18–19`
```typescript
const BOARD_ID = 873;       // 820-02020.bvr
const BINDING_ID = 212;     // existing schematic binding for 873
```
**Issue:** Tests assume board 820-02020.bvr is always ID 873 in the backend databank. If DB is reset or records re-imported, test fails at line 22 `getBinding(page, BINDING_ID)` with silent `undefined` return.

**Severity:** CRITICAL (test suite breaks silently on DB state change)

**Fix:** Scan board list first, look up by filename, then extract ID:
```typescript
async function getBoardIdByName(page, boardName) {
  const res = await page.request.get(`${BACKEND}/api/databank/search?q=${boardName}`);
  const data = await res.json();
  return data.files?.find(f => f.filename.includes(boardName))?.id;
}
```

### 2. Excessive Hard-Coded Ports Conflict with User Environments
**File:** `src/frontend/playwright.config.ts:8, 14`
```typescript
baseURL: 'http://localhost:5174',
command: 'npx vite --port 5174 --strictPort',
```
**Also:** `binding-categorization.spec.ts:17` (backend port 1336), UI tests hardcode 8083

**Issue:** User reported earlier today that port 5174 is used by CRM project. `reuseExistingServer:true` (line 15) means tests may connect to wrong app. If a different Vite server is already running on 5174 (CRM, other project), Playwright tests run against that server silently.

**Severity:** CRITICAL (causes test pollution; earlier "wrong-app login screen" incident)

**Fix:** Read from ENV variables:
```typescript
const port = process.env.VITE_PORT || 5174;
const backendPort = process.env.BACKEND_PORT || 1336;
```

### 3. Excessive `waitForTimeout` Over Proper `waitFor`
**File:** ~100+ instances across tests (comprehensive.spec.ts:178, 220, 300, 400, 500ms sleeps)

**Issue:** Tests use hard-coded `page.waitForTimeout(500)` instead of waiting for actual condition. This is flaky:
- Condition met in 100ms? Still sleep 500ms → slow, wasteful
- Condition not met in 500ms? Flake, then pass → inconsistent

Example (comprehensive.spec.ts:178):
```typescript
await searchInput.fill('');
await page.waitForTimeout(300);  // ← No assertion what we're waiting for
```

**Severity:** IMPORTANT (flakiness vector)

**Better pattern:**
```typescript
await expect(page.getByTestId('search-results')).not.toBeVisible();
```

### 12. Test Setup/Teardown Hygiene Weak
**Instance:** binding-categorization.spec.ts:65–75 creates test binding, deletes after:
```typescript
const created = await create.json();
// ... test assertions ...
await page.request.delete(`${BACKEND}/api/databank/bindings/${created.id}`);
```

**Problem:** If test fails between create and delete, binding leaks into DB. Subsequent runs see different state.

**Better pattern:** Use test.afterEach() or fixtures:
```typescript
test.afterEach(async ({ page }) => {
  if (testBindingId) {
    await page.request.delete(`${BACKEND}/api/databank/bindings/${testBindingId}`);
  }
});
```

### 13. Multi-Board Tab Stability Tests Lack Isolation
**File:** comprehensive.spec.ts:212–249 (multi-board tab switching)

**Issue:** Tests load files, switch tabs, assert stats. But no cleanup between tests. If test-board.bvr tab persists, second test in suite sees it already open.

**Example (line 212):**
```typescript
test('multi-board: opening same file twice reuses tab', async ({ page }) => {
  await page.goto('/');
  await loadBoard(page, TEST_BVR1);
  // Assumes fresh app state; if prior test left a tab, this assertion may differ
```

**Better:** Use `test.beforeEach` to clear state:
```typescript
test.beforeEach(async ({ page }) => {
  await page.goto('/');
  // Close all tabs except the blank initial state
});
```

## Test Quality Assessment

### Idempotence & State Management
- **Positive:** Most E2E tests start with `page.goto('/')` (fresh state)
- **Negative:** binding-categorization.spec.ts tests mutate server state (PATCH /bindings) without transaction rollback
- **Recommendation:** Use SQLite transactions or fixtures to isolate test mutations

### Error Handling Consistency
- **Good:** comprehensive.spec.ts, cross-format.spec.ts filter expected WebGL errors
- **Bad:** binding-categorization.spec.ts, pdf-search.spec.ts lack error handlers; fail silently on network issues
- **Recommendation:** Add consistent `filterExpectedErrors()` to all E2E suites

### Performance & Timeouts
- **Timeout settings:** playwright.config.ts:5 (`timeout: 30000`) is generous; most tests finish in <10s
- **Sleeps:** 100+ `waitForTimeout` calls are avoidable; replace with `waitForSelector`, `waitForNavigation`, etc.
- **Recommendation:** Audit & eliminate hard-coded sleeps; set `waitForNavigation` as default

### Selector Strategy
- **CSS classes used:** 162+ direct class selectors (fragile to CSS refactors)
- **Test IDs:** Good coverage in core tests (e.g., `data-testid="statusbar"`), but missing on panels (library, PDF, debug)
- **Recommendation:** Audit all `.locator('.class-name')` calls; replace with `.getByTestId()` where possible

## Cross-Cutting Observations

### Test Framework Maturity
- **Playwright:** Well-configured, good retry/timeout defaults, screenshot-on-failure enabled
- **Missing:** No unit test framework (vitest/jest) for pure functions; all tests are E2E
- **Recommendation:** Add vitest for parser utilities; keep Playwright for E2E only

### CI/CD Readiness
- **Concern:** Tests require seeded databank (board ID 873), running backend (port 1336), hardcoded ports
- **Result:** Cannot run in standard CI (GitHub Actions, GitLab CI) without Docker Compose setup
- **Recommendation:** Containerize test environment; use services: backend, databank fixtures

### Test Documentation
- **Strength:** Each test file has clear `describe('...')` blocks and descriptive test names
- **Weakness:** No README in tests/ directory explaining fixtures, setup, running tests locally vs CI
- **Recommendation:** Create `src/frontend/tests/README.md` with:
  - How to run locally (`npm run test:e2e`)
  - How to debug (headed mode, slowMo)
  - Fixture setup (which backend, which DB)
  - Known flakes / workarounds

### Technical Debt
1. **Parser format specs exist (docs/formats/)** but test coverage doesn't match; parser tests should reference format specs
2. **CLAUDE.md documents critical PixiJS gotchas** (destroy, BitmapFont.uninstall) but tests don't validate them
3. **Recent feature commits** (v8 migration, binding-categorization, pad-shapes, net-lines 3-state) have tests, but not comprehensive
4. **Library binding flow** is new and complex; tests cover API but not user-facing bind-picker UI

### Test Naming Quality
- **Good:** `test('net lines toggle cycles off → star → chain → off')` clearly states expected behavior (line 327 comprehensive.spec.ts)
- **Good:** `test('BRD: both load without errors')` (cross-format.spec.ts line 55)
- **Bad:** `test('all toolbar buttons render without errors')` (comprehensive.spec.ts line 28) — too vague, doesn't specify which buttons
- **Recommendation:** Use present tense, include expected outcome: `test('clicking net-lines toggle cycles mode from off → star → chain')`

## Conclusion

BoardRipper test suite is **strong in E2E coverage** (multi-format loading, panel switching, net-lines toggle) but **weak in unit tests** (zero pure-function validation, 4 format parsers untested), **backend coverage** (1/4 handlers), and **environmental fragility** (hard-coded IDs, ports, paths, excessive sleeps).

**Immediate actions:**
1. Add BRD parser tests (used in production, zero coverage)
2. Migrate binding-categorization tests to Go (backend API layer)
3. Environment-variable ize ports (5174, 1336, 8083) to avoid user conflicts
4. Replace `waitForTimeout` with `waitFor*` conditions (100+ instances)

**Medium-term:**
- Unit test framework for parsers
- PDF viewer zoom/tier/watermark tests
- History-favorites behavior validation
- Real-pad-shape rendering coverage

**Long-term:**
- Visual regression snapshots
- Performance baseline tests
- Parser error-path coverage (corrupt files)
- Cross-format binding consistency matrix
